P0-1：矩阵汇总列 NaN/undefined 【Bug修复】

现状

renderMatrix()中循环for(let j=0;j<12;j++)循环12次，但comp/push数组只有11个元素(index 0-10)。j=11时comp[11]=undefined, push[11]=undefined，渲染为NaN%/undefined/undefined。

根因

表头写了11个时段列+1个"汇总"列=12列，但代码把12列全部当做时段处理，汇总列的数据从数组第12个元素取(越界)。

修改前(有Bug)

for(let j=0;j<12;j++){
  const comp=d.comp[j],push=d.push[j];
  // j=11时: comp=undefined, push=undefined
  // → NaN%/undefined/undefined
}

修改后

// 11个时段
for(let j=0;j<11;j++){
  const comp=d.comp[j],push=d.push[j];
  ...
}
// 单独计算汇总列
const totalComp=d.comp.reduce((a,b)=>a+b,0);
const totalPush=d.push.reduce((a,b)=>a+b,0);
// 汇总推完率 = 全天总完成/全天总推单

修改内容

renderMatrix()函数：将j<12改为j<11，循环结束后单独计算并渲染汇总列（sum(comp)/sum(push)）。

P0-2：矩阵push/comp时段维度混用 【口径统一】

现状

矩阵中push按"商家推单时间"分时段，comp按"送达时间"分时段，两个时段维度不同，导致：

  • 26个单元格推完率>100%（comp>push）
  • 夜宵1(20-22)最严重：城北站181%、天茂站186%、金涵站188%
  • 夜宵2(22-24)全维度comp=0

根因分析

示例：夜宵1(20:00-22:00)
商家在20:30推单 → push计入"夜宵1" ✓
骑手22:15送达 → comp计入"夜宵2(22-24)" ✗
→ 夜宵1有push无对应comp，夜宵2有comp但push不在此时段

示例：夜宵2(22:00-24:00)
商家23:00推单 → push计入"夜宵2"
骑手次日00:30送达 → comp计入次日"夜宵3" ✗
→ 夜宵2的comp=0，所有单都跨日完成了

统一方案

推荐方案 统一按"商家推单时间"分时段（push和comp都用推单时间）

方案时段维度优点缺点
方案A：按推单时间（推荐）push和comp都按推单时间分推完率=comp/push物理含义清晰；无>100%异常；与推完表口径一致需回溯订单详情重新分时段
方案B：按送达时间push和comp都按送达时间分反映实际运营节奏需回溯订单详情重新分时段

理由：推完率的本质是"推了N单，完成了M单"，按推单时间归集最自然。夜宵2推的单就算次日完成，推完率仍应反映该时段的推单完成情况。

修改范围

需从订单详情重新清洗，将comp也按商家推单时间分时段（而非送达时间）：

  1. MATRIX.data中所有维度的comp数组 → 重新按时段归集
  2. MATRIX.data中所有维度的push数组 → 保持不变（已经按推单时间）
  3. STATION_DATA中cancel/complete/push → 同理重新按时段归集
  4. CROWD_DATA → 同理重新按时段归集

P0-3：夜宵2时段显示异常 【口径统一，同P0-2解决】

夜宵2(22-24)全维度comp=0/push>0，渲染为0.00%。统一口径后（push和comp都按推单时间分），该时段的推完率将恢复正常的88%-98%区间。

此问题随P0-2一并解决，无需单独修改。

P1-4：KPI卡片n/d 与矩阵汇总comp/push 不一致 【口径统一】

现状

指标KPI卡片矩阵汇总差异来源
完成(n/comp)28,87129,665+794KPI←推完表 / 矩阵←订单详情
推单(d/push)29,40631,822+2,416KPI←推完表美配口径 / 矩阵←全部外卖
推完率98.18%93.22%-4.96pp—

根因

  • KPI卡片：数据来自推完表（美配口径），推完率=加盟完成率分子+众包完成率分子 / 推完表美配分母
  • 矩阵汇总：数据从订单详情清洗（全部外卖口径，含跑腿/海葵可能已过滤但统计范围不同）
  • 两个数据源的统计口径不同，导致完成量差794单、推单量差2416单

统一方案

推荐 矩阵数据统一使用推完表口径（与KPI卡片一致）

  1. MATRIX的comp/push数值 → 来自推完表（而非订单详情），确保与KPI卡片的n/d一致
  2. 或：KPI卡片改为使用矩阵汇总数据，但需要额外计算jm_pr/zb_pr等衍生指标

注：订单详情可继续用于STATION_DATA的分时段统计（推完表无分时段数据），但MATRIX汇总列和KPI卡片必须用同一数据源。

P1-5：ot3i/ot3o口径不一致导致趋势断崖 【口径统一】

现状

日期ot3i(3km内)ot3o(3km外)数据来源
7/1 - 7/94.79%~5.33%8.68%~9.28%体验表·全运力绝对超时
7/100.84%2.36%外部看板（口径不明）
7/114.90%8.46%体验表·全运力绝对超时

7/10的值突然降低5倍，7/11又跳回来，趋势图出现明显断崖。

根因

7/10的ot3i/ot3o来自外部看板独立数据源，计算口径与7/1-9、7/11不同（此前多次验证无法从体验表列精确还原）。

统一方案

推荐 所有日期统一用体验表口径计算ot3i/ot3o

  1. 将7/10的ot3i/ot3o改用体验表数据计算（需确认7/10体验表列的精确计算方式）
  2. 如果体验表计算方式仍无法精确匹配，则统一标注"口径：体验表绝对超时"并接受该口径
  3. 目标：消除趋势图断崖，使7/1-7/11的超时率趋势平滑

P2-6：CROWD_DATA与MATRIX众包分时段不一致 【口径统一】

现状

夜宵491.08%106.46%+15.38pp

根因

全天数据完全一致(11505/13258)，但分时段差异巨大。两个数据源的分时段方式不同：
• MATRIX众包：push按推单时间、comp按送达时间（混用）
• CROWD_DATA：push和comp都按推单时间分（统一）

统一方案

随P0-2统一口径后，MATRIX众包的分时段数据也改为"全部按推单时间"，此时CROWD和MATRIX众包的分时段将自动一致。
